# Lab 04 - User Stories and Acceptance Criteria

## User Story Format
As a [user role], I want to [goal/action], so that [benefit/value].

## User Stories

| Story ID | Role | User Story | Related Requirement | Priority | Acceptance Criteria | Demo Evidence |
|---|---|---|---|---|---|---|
| US-01 | Student | As a student, I want to land on a homepage that clearly shows what SmartTimetable does, so that I immediately understand how it helps me find my schedule. | FR-01 | Must | Given I open SmartTimetable, when the homepage loads, then I see the project name, a short description of the problem it solves, and a clear action button (e.g. "View My Schedule"). | Screenshot of homepage |
| US-02 | Student | As a student, I want to move easily from opening the app to viewing my class details, so that I can quickly confirm where and when my next class is. | FR-02 | Must | Given I am on the homepage, when I click "View My Schedule," then I am taken through a clear path: homepage → schedule list → class detail view, with no broken or confusing steps. | Demo walkthrough recording |
| US-03 | Student | As a student, I want to submit a report when I notice a mismatch between the timetable and what actually happened in class, so that the information can be corrected for others. | FR-03 | Must | Given I am viewing a class entry, when I click "Report an issue" and fill in the form, then my report is submitted and stored in the system. | Screenshot of submission form and confirmation |
| US-04 | Admin | As an admin, I want all submitted schedule data and reports to be saved in a database or spreadsheet, so that the information is not lost and can be reviewed later. | FR-04 | Must | Given a student submits a class or a report, when the submission is completed, then the data appears in the connected database/spreadsheet (e.g. Firebase, Google Sheet, or local JSON). | Screenshot of stored data |
| US-05 | Student | As a student, I want to view a list of all my classes for the week, so that I can plan my day without needing to check a separate PDF. | FR-05 | Must | Given I am logged into or using the app, when I open the schedule list, then I see all my classes with time, room, and course name displayed. | Screenshot of class list view |
| US-06 | Student | As a student, I want to search or filter classes by course code or room number, so that I can quickly find specific information instead of scrolling through everything. | FR-06 | Must | Given I am on the schedule list, when I type a course code or room number into the search bar, then only matching results are displayed. | Screenshot/demo of search in action |
| US-07 | Student | As a student, I want to click on a class to see full details, so that I can confirm the exact room, time, and lecturer before heading there. | FR-07 | Must | Given I am viewing the schedule list, when I click on a specific class, then a detail view opens showing the room, time, lecturer, and any reported changes. | Screenshot of class detail view |
| US-08 | Student | As a student, I want to see whether a class is confirmed, changed, or cancelled, so that I know whether to trust the original schedule or expect something different. | FR-08 | Must | Given a class has a status update, when I view that class, then I see a clear status label such as "Confirmed," "Room Changed," or "Cancelled." | Screenshot of status label on class detail |
| US-09 | Admin | As an admin, I want to update the status or details of a class when a change is reported, so that students see accurate, up-to-date information. | FR-09 | Must | Given a student has reported a mismatch, when the admin reviews and updates the class entry, then the new status/details are reflected in the student-facing view. | Screenshot/demo of admin update flow |
| US-10 | Student | As a student, I want the system to prevent me from submitting an incomplete report, so that my report is actually usable for correcting the timetable. | FR-10 | Must | Given I am filling out the report form, when I try to submit with a required field empty, then the system shows an error message and blocks submission. | Screenshot of validation error |
| US-11 | Student | As a student, I want to see a confirmation after submitting a report, so that I know it was received and don't submit it again by mistake. | FR-11 | Must | Given I submit a valid report, when the submission completes, then I see a message such as "Report received" or "Submission successful." | Screenshot of confirmation message |
| US-12 | Admin | As an admin, I want to see a summary view of how many reports or changes have been submitted recently, so that I can quickly identify which classes need attention. | FR-12 | Should | Given reports exist in the system, when the admin opens the dashboard, then they see a summary such as total reports this week or most-reported classes. | Screenshot of admin dashboard/summary |
| US-13 | Student | As a student, I want every screen in the app to look and feel consistent, so that I can navigate the system without confusion. | FR-13 | Must | Given I move between the homepage, schedule list, and detail view, when I view each screen, then the layout, navigation, and button styles remain consistent. | Screenshots across multiple screens |
| US-14 | Student | As a student, I want to use SmartTimetable comfortably on my phone, so that I can check my schedule on the go, not just on a laptop. | FR-14 | Must | Given I open SmartTimetable on a mobile-sized screen, when I navigate the app, then the layout adjusts and remains usable without horizontal scrolling. (Directly supported by Lab 03 evidence: one respondent specifically described the PDF's poor mobile readability, needing to pinch, zoom, and scroll on small screens.) | Screenshot of mobile view |
| US-15 | Student | As a student, I want my submitted report to only ask for necessary information, so that I don't have to share unnecessary personal data. | FR-15 | Must | Given I open the report form, when I view the required fields, then only class-related fields (course, room, issue type) are required, with no unnecessary personal data requested. | Screenshot of report form fields |
| US-16 | Admin | As an admin/team member, I want every feature in the prototype to clearly connect back to a real piece of Lab 03 or Lab 04 evidence, so that we can explain why each feature exists during the demo. | FR-16 | Must | Given a reviewer asks why a feature exists, when the team explains it, then the team can point to the specific user story, requirement, and piece of Lab 03 evidence behind it. | This document plus the traceability shown in mvp-feature-list.md |
| US-17 | Lecturer | As a lecturer, I want to broadcast a schedule change directly through the app instead of using LINE, so that students get the update from one official source. | FR-09 | Could | Given a lecturer needs to announce a change, when they post it through the app, then the change appears immediately in the affected students' schedule view with a clear status label. | Screenshot/demo of lecturer broadcast flow |


## Rejected / Future User Stories
| Story ID | Reason for Postponing | Future Condition |
|---|---|---|
| US-18: As a student, I want to receive push notifications the instant a class changes, so that I don't have to check the app manually. | Strongly requested in Lab 03 evidence, but requires notification infrastructure (e.g. Firebase Cloud Messaging or LINE Notify integration) that adds setup complexity beyond what is needed to prove the core problem-solution fit first. | Add once the core schedule viewer, search, and status-tracking features are working and validated with users. |
| US-19: As a student, I want my class schedule to sync automatically with my personal calendar app, so that I don't need to check SmartTimetable separately. | A few respondents requested calendar sync, but this requires third-party calendar API integration that adds complexity beyond the MVP scope. | Add once the core schedule viewer and reporting features are working and validated with users. |
